1. Goals, non-goals, and success criteria

1.1 Goals

  • Speed as a first-class design constraint. Remove runtime frameworks and third-party render-blocking dependencies. The page should load quickly on a mid-range phone on a weak connection.
  • Legibility-first minimalism. Minimal does not mean sparse; it means every visible element has a job. Typography, spacing, and alignment do the heavy lifting.
  • Subtle cross-cultural influence. Borrow Apple’s clarity discipline (hierarchy, restraint, typographic care) and Japanese aesthetic restraint (ma, emptiness, quiet transitions) without shouting “Apple” or “Japanese Zen”.
  • Academic credibility in the first screen. Make the “proof” visible early: journal placements, current R&R, and fast paths to CV / papers / contact.
  • Accessibility as correctness. Meet WCAG 2.2 AA for contrast, focus, semantics, and keyboard access. Minimalist design often fails here; this spec treats it as a blocker.
  • Maintainability without a build step. A single index.html plus existing assets. CSS and JS should be small, named, and comprehensible.

1.2 Non-goals

  • No blog engine, CMS, or multi-page routing.
  • No heavy animation, parallax, or “portfolio” theatrics.
  • No analytics by default. If added later, it must be optional and privacy-respecting.
  • No auto-fetching of large PDFs (avoid surprise downloads).

1.3 Success criteria (measurable)

  • Dependency budget: zero third-party JS/CSS frameworks at runtime (no Tailwind CDN, no Alpine, no icon fonts).
  • Core Web Vitals target: “Good” for LCP, INP, and CLS in typical Lighthouse mobile runs; track regressions over time.
  • Accessibility target: No WCAG AA contrast failures for text and key UI; keyboard navigation works end-to-end; visible focus indicators in both themes.
  • Content fidelity: No invented publications, affiliations, dates, or links. Preserve paper metadata.

2. Design principles (subtle Apple + Japanese restraint)

2.1 A note on “Apple principles”

Apple’s best interface work is not a “look”; it is a discipline: hierarchy, legibility, and an almost stubborn refusal to compete with content. For this site, we translate that into:

  • Clarity: typographic hierarchy and obvious interaction affordances.
  • Deference: UI supports the content (papers, CV, contact) instead of competing with it.
  • Depth (in moderation): subtle layering only when it improves comprehension (e.g., sticky header legibility), not for “premium vibes”.

Primary sources: Apple design tips and Human Interface guidance (see References).

2.2 A note on Japanese design principles

Japanese aesthetics gives us a powerful warning: when you remove ornament, every remaining decision becomes loud. We use a few concepts as guardrails:

  • Ma (間): “negative space” as an active structural element. Spacing is not leftover — it’s meaning.
  • Shibui / Shibusa: quiet, unobtrusive refinement. No shouting, no gimmicks.
  • Wabi-sabi (applied carefully): allow slight texture and humanity (e.g., serif headings) without slipping into “rustic”.
  • Emptiness (Su / Ku): leave intentional room so the reader can think; avoid filling gaps with icons or decorative rules.

Primary sources: MUJI’s writing on “emptiness” and academic discussions of Japanese minimalism (see References).

2.3 What “high taste” means operationally

  • Few decisions, consistently executed. Limited type sizes, a restrained palette, and a repeatable spacing rhythm.
  • Material honesty. Links look like links. Buttons look pressable. Cards look like grouped content, not “glass panels”.
  • Restraint under pressure. Mobile layouts collapse cleanly; nothing becomes cramped or noisy.

3. Design system: tokens and scales

This site is intentionally small, but it still benefits from a token system. Tokens reduce drift: they keep spacing, color, and typography coherent as content evolves.

3.1 Color tokens (light + dark)

  • --bg: page background.
  • --surface: subtle raised backgrounds (cards/filters) — must remain quiet.
  • --text: primary text color.
  • --muted, --muted-2: secondary text (metadata, labels). Use sparingly; do not create “gray-on-gray” contrast failures.
  • --line: borders and separators. Prefer hairlines to heavy boxes.
  • --accent: one restrained accent used for hover, focus, and the occasional highlight. Must be AA-contrast-safe in both themes.

3.2 Typography tokens

  • Body: system sans stack (fast, crisp, modern). Avoid remote font loads by default.
  • Headings: system serif stack (academic tone, print-like authority).
  • Mono: system monospace for BibTeX/code blocks.

3.3 Spacing scale

Use a small spacing vocabulary (e.g., 0.45rem, 0.7rem, 1rem, 1.6rem, 2.3rem, 3.1rem) and repeat it. In restrained design, irregular spacing is immediately noticeable.

3.4 Radius + shadow

  • Use a single “soft rectangle” radius (10–14px) for cards and panels.
  • Shadows should be subtle and only where they convey grouping or hierarchy. In dark mode, reduce or soften shadows to avoid muddy halos.

4. Component specs

4.1 Header + nav

  • Sticky header with mild backdrop blur for legibility, not decoration.
  • Nav is an anchor list to sections; avoid dropdowns.
  • Primary CTA is the CV link; secondary CTAs include theme toggle.
  • Tap targets: aim for ~44px height for comfortable touch use (especially on mobile).

4.2 Hero (“title page”)

  • Top line (kicker): role + institution.
  • Main heading: a short statement that signals research taste (measurement, structure).
  • Second line: a thesis-like quote for personality (kept short).
  • Immediate actions: Scholar, SSRN, CV, email.
  • Portrait: responsive <picture> with reserved width/height to avoid CLS.

4.3 Credibility strip

A compact “proof” block below the hero. It answers: “Is this person real and active?” in seconds. Keep it factual and calm.

4.4 Selected work cards

  • 3 cards max: the best representatives of the portfolio.
  • Each card includes: status + year, title, authors, one-sentence contribution, and resource pills when available.

4.5 Research list + filters

  • Baseline: papers are readable as HTML, with abstracts hidden behind <details>.
  • Enhancement: JS filters by keyword, status, and topic; updates URL query parameters so filtered views can be shared.
  • Each paper contains: title (primary link), authors, year, status, contribution sentence, topic chips, abstract, resources, BibTeX with copy button.

4.6 Teaching + service

  • Keep teaching compact: one course + student guidance.
  • Service list: factual, single paragraph; avoid long walls of text.

4.7 Contact

  • Make email obvious and copyable.
  • Office address in plain text for easy copying.
  • Links: Scholar and SSRN author page.

5. Performance plan (budgets + measurement)

The previous version of the site used runtime frameworks and third-party CDNs. The fastest optimization is deletion. This redesign is intentionally self-contained.

5.1 Budgets (recommended)

  • HTML: keep under ~60KB uncompressed when possible.
  • JS: keep under ~10KB uncompressed; no third-party JS.
  • CSS: inline in the HTML; keep under ~20KB uncompressed.
  • Requests: avoid external requests in the critical path (fonts, icons, frameworks).

5.2 Core Web Vitals targets

  • LCP (Largest Contentful Paint): aim for “Good” (≤ 2.5s).
  • INP (Interaction to Next Paint): aim for “Good” (≤ 200ms).
  • CLS (Cumulative Layout Shift): aim for “Good” (≤ 0.1).

See web.dev documentation for definitions, thresholds, and measurement guidance.

5.3 Implementation techniques used

  • System fonts (no remote font downloads).
  • Responsive portrait via <picture> + explicit dimensions + fetchpriority for stable LCP.
  • Minimal inline JS (theme toggle, filters, copy BibTeX).
  • Optional content-visibility: auto on long lists to reduce initial render cost in supporting browsers.

5.4 Measurement checklist

  1. Run Lighthouse in Chrome DevTools with mobile emulation; record scores and key opportunities.
  2. Run PageSpeed Insights for field data if the site has sufficient traffic.
  3. Re-test after any significant content change (new PDFs, large images, new sections).
  4. Track regressions: performance should have a “blocker” threshold in the rubric.

6. Accessibility plan (WCAG 2.2 AA)

Minimalist sites often fail accessibility by turning everything into faint gray text, hiding focus rings, and relying on hover. This spec treats accessibility as non-negotiable.

6.1 Required behaviors

  • Keyboard navigation: tab through nav, filters, paper details, and contact without traps.
  • Visible focus: use :focus-visible outlines with strong contrast in both themes.
  • Contrast: ensure text and UI components meet AA thresholds; avoid low-contrast accent colors in dark mode.
  • Reduced motion: respect prefers-reduced-motion; avoid forced smooth scrolling and long transitions.
  • Semantics: landmarks (header, nav, main, section, footer), headings in correct order, labels for form controls.

6.2 Practical test plan

  • Keyboard-only walkthrough (no mouse) on desktop.
  • Screen reader spot-check: nav label, section headings, paper details summaries, button labels.
  • Zoom to 200% and confirm layout reflows without horizontal scrolling for text.
  • Dark mode contrast check, especially for the accent color and muted text.

Primary source: W3C WCAG 2.2 Recommendation.

7. SEO & structured data

  • Keep a descriptive <title> and meta description.
  • Use Open Graph / Twitter cards for previews.
  • Include schema.org Person JSON-LD with affiliation and sameAs links (Scholar, SSRN).
  • Use a canonical URL to avoid duplicate-indexing if mirrored.

8. GitHub Pages deployment notes

  • GitHub Pages serves static files. This design assumes no server-side rendering, no build pipeline, and no dynamic routes.
  • Keep CNAME if using a custom domain.
  • Do not rely on environment-specific paths; use relative links for assets in the repo root.
  • If you later add folders, ensure links remain relative and case-sensitive.

9. Self-hardening rubric (scored, evidence-based)

This rubric is designed to prevent “slow drift”: the gradual degradation that happens when small edits accumulate. The goal is to make quality measurable.

9.1 Scoring

Score each category 0–5. A release is acceptable only if no category is below its minimum threshold. Treat blockers as blockers.

CategoryWeightMinimumWhat “good” looks like
Content fidelity18%5No invented facts; links work; paper metadata preserved.
Performance18%4No runtime frameworks; budgets respected; CWV targets plausible.
Accessibility16%4Focus visible; contrast AA; keyboard flow; reduced motion respected.
Visual system14%4Consistent spacing/type; restrained accent; calm hierarchy.
Usability10%45-second scan passes; primary actions obvious.
Maintainability8%4Small CSS/JS; tokens; comments only where needed.
SEO/metadata8%3Title/desc/OG/schema present and correct.
Spec integrity8%4Spec is actionable; includes debate + decisions; references listed.

9.2 Evidence schema

  • Before/after bundle evidence: record HTML size, number of requests, and whether any third-party scripts exist.
  • Lighthouse evidence: store screenshots and key metrics (LCP/INP/CLS).
  • Accessibility evidence: record contrast checks and keyboard walkthrough notes.
  • Regression log: for every change, log what might regress and how you verified it did not.

9.3 Hardening rules (anti-regression)

  • Never re-introduce runtime CSS generation (e.g., Tailwind-in-browser).
  • Never hide focus outlines globally.
  • Never add third-party scripts without a written justification and a performance check.
  • Keep the hero above-the-fold simple; do not push key links below the fold.

Appendix B — Iteration 2 notes (added to satisfy +100 word growth)

Iteration 2 makes three surgical changes that are easy to miss but hard to unsee once you know to look for them. First, the hero <h1> is now the person’s name. These are small examples of a broader maintenance rule: prefer changes that increase correctness and predictability without adding visual noise.

Appendix C — Iteration 3 notes (added to satisfy +100 word growth)

Iteration 3 is mostly about robustness under partial CSS support. Two small fallback fixes matter for accessibility. First, the focus outline now falls back to outline: 3px solid var(--accent) before the color-mix() refinement. If a browser does not support color-mix(), users still get a visible focus indicator (a WCAG-critical behavior). Second, the sticky header’s fallback background is now var(--bg) rather than a hard-coded light color, so dark mode never accidentally renders a white header in older or unusual environments.
